📡 System overview (software layer)

The NeroviaFlow software stack is a cloud‑native, event‑driven system that ingests real‑time sensor data from field devices (ESP32 + GSM), stores it in a time‑series database, runs detection algorithms, and delivers alerts and dashboards to end‑users. It is designed for low‑latency, high‑reliability, and cost‑effective scaling from 15 to 10,000+ sites.

⚙️ Firmware (ESP32) – core intelligence

🔁 Main loop & power management

Deep sleep 99% of time, wake every 15 min.
Sensor power MOSFET‑controlled to save battery.
GSM activation only during transmission.
Baseline arrays 7 (days) × 96 (slots) stored in RTC memory.

📊 Detection logic (on‑device)

Flow and pressure readings are averaged over the 15‑min window.
Night‑window flag (1‑4 AM) is evaluated locally.
If flow > baseline_flow + threshold AND pressure < baseline_pressure - threshold for 3 consecutive slots → leak confirmed.
Immediate SMS via GSM + store‑and‑forward if network unavailable.

🔧 Over‑the‑air (OTA) updates

Firmware version check every 24h via GSM. Binary delta updates (~20 kB) to minimise data usage.
Fallback to factory firmware in case of corrupted update.

📡 Data transmission format

JSON payload:

{ "site_id": "CAS‑HOTEL‑001", "flow_lph": 1240.5, "pressure_bar": 3.87, "acoustic_energy": 0.23, "day": 2, "slot": 47, "baseline_ready": true, "battery_voltage": 3.9, "timestamp": "2026‑04‑22T02:15:00Z" }

🧠 Leak detection algorithm – multi‑sensor confirmation

Baseline establishment

After 30 days, for each (day_of_week, slot) the system stores:
• μ_flow (mean) and σ_flow (standard deviation)
• μ_pressure and σ_pressure
• Optional acoustic baseline μ_acoustic, σ_acoustic.
Thresholds are dynamic: threshold_flow = μ_flow + 2 × σ_flow (approx 95th percentile).

Night‑window detection (core)

During 1‑4 AM, every 15 min the device sends data. The cloud engine:

  1. Compares flow, pressure against baselines.
  2. If flow deviation > threshold AND pressure drop > threshold → leak suspect.
  3. Increment consecutive counter. After 3 consecutive suspects → leak confirmed.
  4. Insert alert record, send SMS via Twilio / GSM gateway.

Acoustic enhancement (optional)

When acoustic sensor is present, a third condition is added: acoustic energy > baseline + threshold. A leak is confirmed only when at least two of the three sensors indicate anomaly (flow+pressure or flow+acoustic or pressure+acoustic). This eliminates false positives from silent leaks.

False‑positive mitigation

• Consecutive readings requirement (3).
• Pressure drop must be sustained (not a spike).
• Baseline is updated weekly (moving average) to adapt to seasonal changes.
• Manual override: facility manager can mark an alert as false; that event is fed back to refine thresholds.

🧩 Core UI components

📊 Live charts (Chart.js)

Flow (L/h) and pressure (bar) displayed as interactive line charts. Baseline shown as dashed line. Colour‑coded zones (green = normal, orange = suspicious, red = leak confirmed). Tooltips on hover show exact values and timestamps.

⚠️ Alert cards

Each alert appears as a card with priority colour, timestamp, leak type (flow anomaly, pressure drop, acoustic), and action buttons ("View details", "Mark resolved"). Dismissible after review. High‑priority alerts also trigger a modal with guided troubleshooting steps.

📋 Monthly report generator

Button "Generate PDF" opens a modal with date range selection, site selection (single hotel or group), and options to include ESG metrics, water savings, and CO₂ equivalent. Preview before download. Background generation for large portfolios.

🔧 Device health panel

Compact widget showing battery voltage, GSM signal strength, last transmission time, and firmware version. Colour indicators: green (healthy), orange (attention needed), red (offline). Click to open detailed device log.

🛠️ Admin controls

Buttons for OTA firmware update, device reboot, remote calibration, and SIM management. Confirmation modal required for destructive actions. Bulk operations (e.g., update multiple devices) available for enterprise users.

🌙 Dark mode (planned)

Automatic switching based on system preference. Dark palette: navy background, light text, muted alert colours. Implemented via CSS custom properties and class toggle.

📐 Responsive & accessibility (WCAG 2.1 AA)

📱 Responsive behaviour

Desktop‑first design, but collapses gracefully on tablets and phones.
Tablet charts resize, sidebar becomes collapsible.
Mobile bottom navigation bar, charts stacked vertically, modals full screen.
Touch‑friendly: tap targets ≥44px, swipe gestures for alerts.

♿ Accessibility features

• Colour contrast ratio ≥4.5:1 for text against background.
• ARIA labels on all interactive elements.
• Keyboard navigable: Tab, Shift+Tab, Enter, Escape.
• Focus indicators visible (outline).
• Screen reader friendly: headings, landmarks, alt text for charts.

🌐 Localisation (i18n)

Interface strings in English and French, detected via browser language or user toggle. Dates and numbers formatted according to locale (MM/DD vs DD/MM). RTL support not required for now.

⏱️ Performance targets

First Contentful Paint < 1.5s, Time to Interactive < 3s. Charts lazy‑loaded. Data tables paginated. Asset caching via service worker for offline fallback.

🧭 User flow – Hotel facility manager
1. Facility manager receives SMS alert about a potential leak → clicks link to open dashboard (auto‑login if session active). 2. Dashboard shows red alert card with leak details (flow deviation, pressure drop, time). 3. Manager clicks "View details" → sees live charts with the leak window highlighted. 4. Follows guided troubleshooting (e.g., "Is the cooling tower running?" / "Check zone valve"). 5. After repair, marks alert as resolved and enters notes (e.g., "replaced toilet flapper"). 6. System automatically verifies that night flow returns to baseline → closes alert. 7. At month end, manager generates an ESG report (PDF) to share with hotel ownership, showing water saved and CO₂ avoided.

🔒 Security & compliance

Data in transit

• TLS 1.3 for all HTTPS connections.
• Device ↔ Cloud: mutual authentication using pre‑shared API keys (rotated monthly).
• GSM bearer: encrypted GPRS (operator‑grade).

Data at rest

• AES‑256 encryption for database at rest.
• Client raw data never leaves EU region (Supabase EU).
• Anonymisation pipeline for data monetisation: irreversible aggregation (no site‑level identifiers).

Compliance

• Moroccan Law 09‑08: data processing agreement with each client, CNDP declaration.
• GDPR‑style: right to erasure, data portability.
• Audit logs of all data access.

Backup & disaster recovery

• Daily automated backups (7‑day retention).
• Cross‑region replication for critical metadata.
• RTO < 4 hours, RPO < 1 hour.

💸 Data monetisation pipeline (anonymisation & aggregation)
Raw readings (site‑level) → Anonymisation service (remove site_id, randomise timestamps) → Aggregation by region, hotel size, month → Dataset for utilities/researchers → API access / flat file export.

Aggregated datasets are sold to water utilities (LYDEC, ONEE) for infrastructure planning, to researchers for academic studies, and to infrastructure funds for risk assessment. Pricing: $30/site/month starting from 2028 (50+ sites).
